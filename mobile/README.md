# mobile — 移动端 App（uni-app / HBuilderX）

> 本目录是给「普通用户」用的手机 App。包含两个角色：顾客端（点餐）和骑手端（送餐）。这也是唯一需要调用手机硬件（GPS、相机、麦克风）的部分。

---

## 一、这个文件夹是干什么的？

`mobile` 存放**手机 App** 的全部代码。我们用老师指定的 **HBuilderX** 编辑器 + **uni-app** 框架来开发，写一套代码就能打包成安卓 APK，再放到 **BlueStacks**（安卓模拟器）里运行和演示。

为什么用 uni-app？因为它「一套代码、多端运行」，对新手友好，而且 HBuilderX 把开发、调试、打包都集成好了，不用自己搭复杂的安卓环境。

打个比方：
- 顾客端 = **美团/饿了么用户版**（看菜单、下单、催单）
- 骑手端 = **骑手版 App**（接单、导航、拍照确认送达）

---

## 二、它在整个项目架构里的位置

```
┌──────────────────────────────┐
│ mobile/  ← 你在这里            │
│ uni-app (HBuilderX 开发)       │
│ ┌────────────┐ ┌───────────┐  │
│ │ 顾客端      │ │ 骑手端     │  │
│ │ 点餐/下单   │ │ 接单/送达  │  │
│ └────────────┘ └───────────┘  │
│   ↑ 调用手机硬件：GPS/相机/麦克风 │
└───────────────┬──────────────┘
                │ HTTP 请求 (JSON)
                ▼
      ┌───────────────────┐
      │   backend/ (后端)   │
      └─────────┬─────────┘
                ▼
      ┌───────────────────┐
      │   MySQL 数据库      │
      └───────────────────┘
```

**两个独立的「硬件」要点：**
1. **App 本身不存核心数据**，一切通过 backend 的 API 收发。
2. **硬件调用是本目录独有的职责**，这是评分明确要求的「使用 GPS/相机/麦克风 等硬件设备生成数据」。

## 三、这里面应该放什么（推荐目录结构）

HBuilderX 新建 uni-app 项目后，结构大致如下：

```
mobile/
├── README.md            # 本文件
├── manifest.json        # App 配置：名称、图标、需要申请的硬件权限
├── pages.json           # 页面路由配置：有哪些页面、底部导航等
├── main.js              # 程序入口
├── App.vue              # 根组件
├── pages/               # 一个个页面
│   ├── customer/        #   顾客端页面
│   │   ├── menu.vue           # 浏览菜单
│   │   ├── order.vue          # 下单 / 语音搜索
│   │   └── orderStatus.vue    # 查看订单进度
│   └── rider/           #   骑手端页面
│       ├── taskList.vue       # 待接订单列表
│       ├── navigation.vue     # GPS 定位 / 导航
│       └── delivered.vue      # 拍照确认送达
├── api/                 # 统一封装对 backend 的请求
└── static/              # 图片、图标等静态资源
```

## 四、这里负责实现哪些评分要求

| 评分要求 | 在 mobile 端如何实现 |
| --- | --- |
| App 客户端（安卓） | 整个 mobile 目录，打包成 APK |
| 多角色不同界面 | 顾客端 / 骑手端 两套界面 |
| 登录、下单、接单、完成 | 各页面对接 backend 接口 |
| **硬件 - GPS** | 骑手端用 `uni.getLocation` 获取定位 |
| **硬件 - 相机** | 骑手端用 `uni.chooseImage` 拍照确认送达 |
| **硬件 - 麦克风** | 顾客端用 `uni.getRecorderManager` 做语音搜索 |
| 界面美观、便捷 | uni-app 组件 + 统一 UI 风格 |

> 三种硬件能力是评分硬指标，**至少要实实在在用上一种并能演示**，方案里规划了三种全用，分散在两个角色里。

## 五、新手怎么开始（Phase 3 才正式动手）

1. **下载安装 HBuilderX**（老师指定工具，去 dcloud.io 官网下 Mac 版）。
2. **下载安装 BlueStacks**（安卓模拟器，用来跑 APK）。
3. 在 HBuilderX 里：`文件 → 新建 → 项目 → uni-app`，选一个带 TabBar 的模板起步。
4. 点工具栏的「运行」可以在浏览器里预览基本页面；要测硬件功能，则需「发行 → 打包 APK」后装进 BlueStacks。

**演示前必须提前测通的硬件坑（非常重要）：**
- **GPS**：用 BlueStacks 自带的「定位」工具模拟位置，不要指望模拟器有真 GPS。
- **相机 / 麦克风**：要在 BlueStacks 设置里把电脑的摄像头、麦克风「映射」给模拟器，否则 App 调用会失败。**一定要提前一两天测，别等答辩当天才发现调不起来。**

> 现在是 Phase 0，本目录暂时只放这份 README。移动端是 Phase 3 的任务，依赖 backend 的接口先就绪（或用 Mock）。
