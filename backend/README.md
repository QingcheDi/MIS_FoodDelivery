# backend — Python 后端服务

> 本目录是整个系统的「大脑」。所有移动端 App 和 Web 管理端看到的数据、做的操作，最终都由这里的代码处理。

---

## 一、这个文件夹是干什么的？

`backend` 存放**应用服务器**的全部代码。它是一个用 Python 写的程序，运行后会在你的电脑上开一个「服务」（通常监听 `http://localhost:8000`），等待别人来请求数据。

打个比方：
- 移动端 App、Web 管理端 = **顾客**（提出需求：「给我菜单」「我要下单」）
- backend = **餐厅后厨 + 收银台**（接收需求、处理、查数据库、返回结果）
- 数据库（MySQL）= **仓库**（真正存放数据的地方）

顾客（前端）从不直接进仓库（数据库），所有事情都通过后厨（backend）来办。这样做的好处是：安全（顾客拿不到仓库钥匙）、统一（所有规则都在一处）。

---

## 二、它在整个项目架构里的位置

```
┌─────────────┐     ┌─────────────┐
│ 移动端 App   │     │ Web 管理端   │
│ (mobile/)   │     │ (web/)      │
└──────┬──────┘     └──────┬──────┘
       │                   │
       │   HTTP 请求 (JSON) │
       └─────────┬─────────┘
                 ▼
       ┌───────────────────┐
       │  backend/  ← 你在这里 │
       │  Python (FastAPI)  │
       │  - 验证身份         │
       │  - 处理业务逻辑      │
       │  - 读写数据库        │
       └─────────┬─────────┘
                 │  SQL
                 ▼
       ┌───────────────────┐
       │  MySQL 数据库 (db/)  │
       └───────────────────┘
```

**关键概念：API（接口）**。前端和后端之间靠「API」沟通。一个 API 就是一个约定好的「网址 + 规则」，例如：
- 前端访问 `POST /api/orders` 并带上菜品信息 → 后端就帮它创建一个订单
- 前端访问 `GET /api/menu` → 后端返回所有菜品列表

这些 API 的完整清单写在 `docs/api_contract`（API 契约文档）里，**后端要严格按照那份文档来实现**，前端才能对接上。

## 三、这里面应该放什么（推荐目录结构）

随着开发推进，这个文件夹大致会长成这样（新手前期不用一次建全，按需添加）：

```
backend/
├── README.md            # 本文件
├── requirements.txt     # 项目依赖清单（用了哪些第三方库）
├── .env.example         # 环境变量示例（数据库密码等敏感信息的模板）
├── main.py              # 程序入口，运行它就启动服务器
├── database.py          # 连接数据库的配置
├── models/              # 数据模型：用 Python 类描述数据库的每张表
│   ├── user.py
│   └── order.py
├── routers/             # 各个 API 接口的具体代码，按功能分文件
│   ├── auth.py          #   登录、注册、找回密码、两步验证
│   ├── orders.py        #   下单、接单、更新订单状态
│   ├── menu.py          #   菜单查询
│   └── upload.py        #   上传配送照片 / 语音
├── core/                # 通用工具
│   ├── security.py      #   密码加密 (bcrypt)、JWT 令牌
│   └── deps.py          #   权限校验等公共依赖
└── uploads/             # 上传的图片/语音实际存放处（已在 .gitignore 排除，不上传到 GitHub）
```

> ⚠️ 注意：`.env`（真实密码）和 `uploads/`（用户上传的文件）已在项目根目录的 `.gitignore` 里被排除，**不会**也**不应该**上传到 GitHub。`.env.example` 只是个不含真实密码的模板，用来告诉队友需要配哪些变量。

## 四、这里负责实现哪些评分要求

对照项目要求，后端是「重灾区」，承担了大量核心功能：

| 评分要求 | 在后端如何实现 |
| --- | --- |
| 用户登录、下单、接单、完成 | `routers/auth.py`、`routers/orders.py` 里的接口 |
| Python 应用服务器 | 整个 backend 就是 |
| 密码加密存储 | `core/security.py` 用 bcrypt，绝不存明文 |
| 两步验证、密码找回、微信登录 | `routers/auth.py` |
| 非结构化数据存储 | `routers/upload.py`：文件存磁盘，数据库只存路径 |
| 防 SQL 注入、敏感字段脱敏 | 全程用参数化查询，返回数据时屏蔽敏感字段 |
| AI 推荐 / 销量预测接口 | 后期新增 `routers/ai.py` |

## 五、新手怎么开始（Phase 2 才正式动手，这里先有个印象）

1. **建虚拟环境**（隔离项目依赖，不污染系统 Python）：
   ```bash
   cd backend
   python3 -m venv venv
   source venv/bin/activate    # Mac 激活虚拟环境，成功后命令行前面会出现 (venv)
   ```
2. **安装框架**：
   ```bash
   pip install fastapi uvicorn
   ```
3. **写一个最简单的 main.py 测试**，然后运行：
   ```bash
   uvicorn main:app --reload
   ```
4. 浏览器打开 `http://localhost:8000/docs` —— FastAPI 会**自动生成**一个可点击测试的 API 文档页面，非常适合新手。

> 现在是 Phase 0，本目录暂时只有这份 README 即可。真正写代码是 Phase 2 的事，到时按 `docs/api_contract` 里定好的接口逐个实现。
